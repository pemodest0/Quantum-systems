# Quantum Systems Project Overview

This repository is the working research lab for open quantum systems, with two
tracks kept in the same codebase but separated by module, report and workflow.

## Track A: Open Quantum Control and NMR

Purpose: maintain the original validated base for dissipative dynamics,
tomography, noise spectroscopy, GRAPE control and paper reproduction.

Main paths:

- `src/oqs_control/`: reusable control, tomography and open-system utilities.
- `scripts/run_experimental_decision_pipeline.py`: synthetic experimental workflow.
- `docs/papers/`: paper guides and figure annotations.
- `outputs/repro/`: reproducible paper figures and metrics.
- `reports/review_article/`: review-style PDF and source.

Status: useful as scientific foundation and literature base. It should remain
stable unless a change is directly useful for validation, figures or article
generation.

## Track B: Open Quantum Transport

Purpose: MSc-oriented project on coherent and dissipative transport in small
open quantum networks, with graph topology, disorder, dephasing, loss and sink
efficiency as the main objects.

Main paths:

- `src/oqs_transport/`: core simulation, graph/network, medium, campaign and
  analysis code.
- `configs/transport_*.json`: reproducible campaign configurations.
- `scripts/run_transport_*.py`: one-command experiments and campaigns.
- `scripts/build_transport_*.py`: report, tutorial and figure-pack builders.
- `reports/famb_msc_transport_proposal/`: current MSc proposal PDF/TEX.
- `reports/transport_medium_campaign_report/`: current medium/dephasing campaign.
- `reports/transport_graph_collection_report/`: graph/topology campaign report.
- `outputs/transport_networks/`: generated figures, tables and raw metrics.
- `tests/test_transport_*.py`: regression tests for the transport layer.

Status: active research track and best candidate for the 2-year MSc project.

## Collaboration With Ygor

Keep the repository under Pedro's GitHub account, but separate collaboration in
practice by using:

- issues or notes for open questions and decisions;
- reproducible scripts instead of ad-hoc notebooks;
- reports in `reports/` for results that can be cited in meetings;
- commits grouped by scientific unit: model, campaign, report, validation.

Suggested working rule: if a result cannot be regenerated by one script in
`scripts/` and one config in `configs/`, it is not yet clean enough to use in the
proposal or in a paper.

## Commit Policy

Use small scientific commits:

- `transport core`: package, configs, tests.
- `transport campaigns`: run scripts and generated tables/figures.
- `transport reports`: PDFs/TEX and final figures.
- `docs`: project roadmap, paper notes, lab manual.
- `control stack`: review article and NMR/control reproduction updates.

Avoid mixing temporary files, cache directories and exploratory scratch output
with proposal-facing artifacts.
